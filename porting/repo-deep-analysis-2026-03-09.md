# Kernel-Xiaomi-Umi 深度分析（2026-03-09）

## 1) 仓库定位

该仓库是 **“内核移植编排器”**，而不是完整内核源码仓：
- 核心价值在 CI 工作流 + 迁移脚本链路
- 输入：4.19 源仓 + 5+ 目标仓
- 输出：可诊断的构建工件、DTB 匹配分析、AnyKernel 候选包

当前结构（高层）:
- `.github/workflows/`：编排入口
- `tools/porting/`：迁移与诊断脚本
- `porting/`：计划、基线、盘点、变更记录

## 2) 已有能力评估

### 优势
1. **自动化完整度高**：从迁移、构建、诊断到汇总工件一条龙。  
2. **观测性做得好**：新增了错误摘要、工件索引、metrics、校验、checksum。  
3. **失败可追踪**：大多数步骤 `if: always()`，便于失败后定位。
4. **并发控制存在**：同 ref 运行互斥，减少资源浪费。

### 风险点
1. **工作流单体偏大**：`phase2-port-umi.yml` 责任过于集中，后期维护成本会升高。  
2. **容错模式偏宽松**：大量 `|| true` 会掩盖“应失败”的质量门槛。  
3. **AnyKernel 产物仅候选**：当前可用于诊断，不等于可直接稳定刷入。  
4. **设备筛选策略硬编码**：`phase2_apply.sh` 内规则集中，扩展到多机型时复杂度上升。

## 3) 代码与流程细化观察

### workflow 层
- `timeout-minutes: 90` 已设置，合理。
- 构建失败会保留日志并最终 fail，逻辑正确。
- 目前没有明确的“质量门禁分级”（如 hard fail vs soft warn）。

### phase2_apply.sh
- `set -euo pipefail` 已开启，基础可靠。
- include-aware BFS 迁移思路正确，适合最小可启动迁移。
- 设备关键词策略虽有效，但建议后续外置配置化（便于版本化管理）。

### Python 诊断脚本链
- 覆盖了完整工件生命周期（收集、摘要、结构化导出）。
- 适合后续接入 dashboard 或趋势监控。

## 4) 重整仓库建议（分两阶段）

### 阶段 A（立即可做，低风险）
1. 为 `tools/porting/` 增加 README（已完成）说明脚本职责与执行顺序。  
2. 在根 README 增加“文档入口”与“诊断阅读路径”。
3. 明确“诊断候选包”与“可刷机包”的边界说明，降低误用风险。

### 阶段 B（推进中，中风险）
1. 把 `phase2-port-umi.yml` 拆成复用组件（composite action / reusable workflow）。
2. 引入质量门禁开关：
   - 关键文件缺失 → hard fail
   - 次要诊断缺失 → warn
3. 机型匹配规则配置化（YAML/JSON），减少脚本内硬编码。

## 5) 下一步执行优先级（建议）

P0：README 文档入口重整（当天可完成）  
P1：工作流中“软/硬失败”分层（本周）  
P2：迁移规则配置化（本周~下周）

## 6) 结论

仓库方向正确，已经具备“可持续迭代的移植编排骨架”。当前最应推进的是：
- **先整理可读性与规则边界（降低协作成本）**
- **再提升门禁策略（降低误判与假阳性成功）**

这会比继续堆脚本更快提升整体交付质量。
